# Per-CPU Landing Pages

## Intent

Generate a stable, indexable page per CPU model that appears on Hetzner's auction (e.g. `/servers/cpu/ryzen-5-3600`, `/servers/cpu/xeon-e3-1245v2`). These pages aggregate current listings and 90-day price history for a single CPU into one URL that can earn ranking on long-tail queries like "cheap Ryzen 5 3600 server" or "Xeon E3-1245v2 price".

This is the "Phase B" follow-up flagged in `specs/configurations-rework.md`. It's the largest remaining SEO lever before per-Hetzner-model pages (AX41, EX44 — separate spec, depends on data enrichment we don't have yet).

## Goals

- Each page is genuinely useful to a human searcher: current cheapest config, price history chart, common variants, deep links to filtered `/analyze`.
- Each page emits truthful structured data: `Product` + `AggregateOffer` (lowPrice, highPrice, offerCount, priceCurrency=EUR), plus `BreadcrumbList`.
- Pages are server-rendered for SEO (no client-side hydration delay before content), with edge caching to keep response times fast.
- All generated pages appear in the sitemap.
- An index at `/servers/cpu/` lists every page for crawlability.

## Non-goals

- Per-Hetzner-model pages (AX41, EX44…) — those need product-name enrichment we don't have today.
- Per-configuration pages (`/servers/config/<slug>`) — also out of scope; later spec.
- Anything client-side beyond the existing chart components.

## URL scheme

- `/servers/cpu/` — alphabetical index of all CPU pages, grouped by vendor (Intel / AMD).
- `/servers/cpu/[slug]` — detail page for one CPU.

### Slug rules

Generated by a single helper used in both directions (URL → CPU lookup, CPU → URL build):

1. Strip leading vendor (`AMD `, `Intel `, `Intel(R) `).
2. Strip registered/trademark glyphs (`®`, `™`, `(R)`, `(TM)`).
3. Strip dual-CPU prefix `2x ` and `2 x `.
4. Lowercase.
5. Replace any run of non-alphanumerics with a single hyphen.
6. Trim leading/trailing hyphens.

Examples:

| CPU string                 | Slug              |
| -------------------------- | ----------------- |
| `AMD Ryzen 5 3600`         | `ryzen-5-3600`    |
| `Intel Xeon E3-1245v2`     | `xeon-e3-1245v2`  |
| `2x Intel Xeon E5-2680 v4` | `xeon-e5-2680-v4` |
| `AMD EPYC 7502P`           | `epyc-7502p`      |

The same slug may map to multiple `cpu` strings (e.g. dual-CPU variants). The page covers all of them together — that's the right user model.

## Eligible CPUs

A CPU is eligible for a page only if:

- It has at least one row in `current_auctions` (so the page has live content).
- `cpu_multicore_score` is non-null in that row (no enrichment gap).
- Slug is unique after the rules above (collisions are logged and the lower-listing-count CPU is dropped to a warning — should not happen in practice).

Cap initially at any CPU meeting the above. Expect ~50–100 pages.

## Page content (`/servers/cpu/[slug]`)

In rendering order:

1. **H1**: "Hetzner servers with `<CPU name>`" (full canonical name from `cpu` field).
2. **Lead paragraph**: cores / threads / generation / Geekbench 5 multicore score / cheapest current price.
3. **Cheapest live listing** (one card, full server config, link to `/analyze` with filter pre-applied to that CPU).
4. **Price history chart** — daily minimum price for this CPU over the last 90 days. Reuse statistics-page chart components if possible.
5. **Common variants** — top 5 (cpu, ram_size, storage_shape) groupings ranked by listing count over the last 30 days, each with cheapest current price and "Browse →" deep link.
6. **Related CPUs** — sidebar/strip with 3–4 same-vendor or same-generation CPUs, linking to their pages.
7. **Footer link** back to `/configurations` and `/servers/cpu/` index.

## Schema.org

Per page:

- `BreadcrumbList`: Home → Servers → CPU → `<CPU name>`.
- `Product` for the CPU itself, with `name`, `description`, `category: "Dedicated Server"`, `brand: { @type: Organization, name: "Hetzner" }`, `additionalProperty` array for cores/threads/score/generation.
- `AggregateOffer` nested under the Product, with `lowPrice` (cheapest current), `highPrice` (most expensive current), `priceCurrency: "EUR"`, `offerCount` (live listings).
- Optionally one full `Offer` for the cheapest listing (matches what's on screen).

## Index page (`/servers/cpu/`)

- H1: "Browse Hetzner servers by CPU".
- Two sections: AMD and Intel.
- Inside each section, alphabetical list of `<a href>` to each CPU page, with a small `<dim>` showing live listing count.
- Add `BreadcrumbList` and an `ItemList` schema enumerating CPU pages.

## Sitemap

Extend `src/routes/sitemap.xml/+server.ts` to pass `paramValues` from `super-sitemap`:

```ts
const slugs = await loadCpuSlugs();
return await sitemap.response({
  origin: "https://radar.iodev.org",
  paramValues: { "/servers/cpu/[slug]": slugs },
});
```

`loadCpuSlugs()` queries the eligible-CPU set and returns the slug list. Cache the result in module scope for the sitemap response — super-sitemap is hit infrequently.

## Data layer

New `src/lib/api/shared/cpu-pages.ts` (shared between page server loaders and sitemap):

- `slugifyCpu(cpu: string): string`
- `loadEligibleCpus(db): Promise<{ slug: string; cpus: string[]; vendor: 'AMD' | 'Intel' }[]>` — returns the list of eligible slugs with the full set of `cpu` strings each represents.
- `loadCpuPageData(db, slug): Promise<CpuPageData | null>` — returns everything the detail page needs (current cheapest, listing count, price-history daily mins, common variants, related CPUs).

Queries against D1 (`current_auctions` for current state, `auctions` for history). Use `cpu IN (?, ?, ...)` to handle slug-to-multiple-CPU mappings.

## Caching & rendering

- SSR via SvelteKit page server load.
- Set response headers in load: `cache-control: public, max-age=300, s-maxage=3600, stale-while-revalidate=86400`.
- Cloudflare Pages caches at the edge based on these headers; users get fresh-enough content (≤1h stale) at static-page speed.
- Do **not** prerender at build time — D1 isn't available during build.

## Open question (one)

**Slug collisions.** "AMD EPYC 7502" and "AMD EPYC 7502P" produce different slugs (`epyc-7502` vs `epyc-7502p`), so that's fine. "Intel Xeon Gold 6132" and "Intel Xeon Silver 6132" — both real CPUs, both real slugs, no collision either. I think collisions are unlikely in practice, but the implementation will log + drop on collision, never silently pick. Flag if this turns out to be wrong.
